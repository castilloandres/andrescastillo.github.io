Why use immutable vs mutable code?

23/02/2019

By: Andres Castillo Ormaechea

I am creating this post because I was using a simple for loop that drove me into a rabbit hole for two (2) days. So I thought I should at least dedicate a small post to explain how to avoid this from the very beginning.

But first. What the heck is mutable and immutable code?

The word mutable refers to "liable to change" or simply put "changeable". When talking about mutable code we are referring specifically to mutable data structures like objects -> {} , arrays -> [] etc. and there possibility of changing its form or structure. An example to illustrate mutable and immutable code:

Let's say our friend Ramón wants to change their name to Francisco. So we will do it first with mutability:

> let ramon = {
    name: 'Ramón',
    Surname: 'Melao',
    age: 21
  };

// Here we change Ramon's name a.k.a mutate to "Francisco"
> ramon.name = 'Franciso';

> { name: 'Franciso', Surname: 'Melao', age: 21 }

Now, lets do the same with immutability:

> const ramon = {
    name: 'Ramón',
    Surname: 'Melao',
    age: 21
  };

// Here we create a new object with Ramon's new name -> Francisco.
const { name, ..rest } = ramon;
> const francisco = {
  name: 'Francisco',
   ...rest
};

> { name: 'Franciso', Surname: 'Melao', age: 21 }

Can you notice the difference between the two? We can see here that instead of changing an object property of ramon we decided to create a new object that will incorporate the changes we wanted to make in the name.

Ok, I see... But what is the point? How can this benefit me?

When we write mutable code we are making changes to data structures that already exist and in turn there is a higher possibility of clashing pointers when using variables in the mix which brings up unexpected behaviour in all sorts of ways and that is hard to debug when not being aware of a variable pointing to two different objects (clashing pointers) that in the end takes you to a rabbit hole (somewhere you don't want to be).

On the other hand when we use immutable code there is no possibility for clashing pointers since we are always creating new data structures instead of modifying and re-using old ones.
